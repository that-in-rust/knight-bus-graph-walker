# Knight Bus v001 ELI5

## Big Idea

The big decision after the last thesis commit is simple:

**for `v001`, stop trying to prove everything at once and prove one narrow thing well.**

That narrow thing is:

- start with `nodes.csv` and `edges.csv`
- compile them into a graph snapshot
- prove the snapshot gives the same answers as the CSV truth
- query the snapshot from a CLI
- benchmark the snapshot honestly

This note explains what changed after commit `657d23f` and why the plan got smaller on purpose.

## Why This Matters

The earlier thesis was about the **shape of the bytes**.

It said:

- do the heavy graph work at build time
- keep the runtime cheap
- store forward and reverse adjacency in a walk-friendly format
- avoid turning the runtime into a database query engine

That part is still true.

But a hackathon has a different problem:

- too many moving parts can kill the demo
- too much ambition can hide whether the core idea actually works

So the newer discussion answered a more practical question:

**what is the smallest believable demo that still proves the core storage idea?**

## What Changed After The Last Commit

Commit `657d23f` established the storage-runtime thesis.

After that, we narrowed the execution plan.

Before, the conversation included:

- large internet datasets
- synthetic generation
- possible HTTP serving
- direct streaming ingest
- a bigger “full system” story

After slowing down, the decision became:

- `v001` assumes `nodes.csv` and `edges.csv` already exist
- `v001` is CLI-only
- `v001` does not own dataset import
- `v001` does not own synthetic generation
- `v001` does not own HTTP
- `v001` does not own Wasm packaging

In plain English:

we moved from “build the whole city” to “build the strongest first bridge.”

## Core Ideas Made Simple

### 1. CSV Is The Truth, Snapshot Is The Machine

Think of the CSV files like the teacher's answer sheet.

They are useful because they are:

- readable
- inspectable
- easy to compare against

But they are not how we want to answer fast graph questions.

The snapshot is the machine-built version.

It is shaped for fast graph walking:

- dense IDs
- forward adjacency
- reverse adjacency
- direct offset jumps

So the rule is:

- CSV tells us what is correct
- snapshot gives us speed

That is why the new PRD keeps both layers instead of pretending one file format can do every job well.

### 2. Small Proofs Make Big Claims Trustworthy

Jumping straight to a `20+ GiB` run sounds bold, but it is risky.

If the giant run fails, we do not know whether the problem was:

- the CSV input
- the build pipeline
- the snapshot format
- the query logic
- the benchmark logic
- the machine limits

That is why we chose a proof ladder:

1. tiny CSV fixture
2. medium CSV fixture
3. large CSV fixture

It is like checking a new oven with a small cake before baking food for a whole wedding.

The recipe stays the same.

Only the batch size changes.

### 3. CLI-Only Is A Strength, Not A Shortcut

HTTP is useful later, but it is not the main proof.

For `v001`, the real proof is:

- can we build the snapshot?
- can we verify parity?
- can we query it?
- can we benchmark it?

A CLI is enough for that.

Adding HTTP too early would add:

- server lifecycle problems
- endpoint contracts
- more failure modes
- more test surface

That is extra furniture before the floor is finished.

So `v001` is deliberately:

- `build`
- `verify`
- `query`
- `bench`

HTTP can come in `v002` once the runtime is already trustworthy.

### 4. The Honest Claim Is About Working Set, Not Magic

One important correction came out of the discussion.

The right claim is **not**:

- “we use zero RAM”
- “everyone else must load the whole graph”

The honest claim is:

- the compiled snapshot can be larger than RAM
- the runtime touches only the pages it needs for the query
- hot query RSS should stay below full snapshot size

That is the important proof.

A good analogy is a library.

You can use a huge library without carrying the whole building in your backpack.

You just open the shelves and pages you need right now.

## What v001 Actually Proves

`v001` should be able to answer these questions clearly:

- Can `nodes.csv` and `edges.csv` be compiled into a forward-plus-reverse walk snapshot?
- Does the snapshot return the same neighbors as the CSV truth layer?
- Can a CLI query one-hop and two-hop neighborhoods from the snapshot?
- Can the same pipeline scale from tiny CSVs to a large snapshot artifact?
- Can the benchmark record snapshot size, peak RSS, and latency without mixing up the story?

This is enough to prove the central runtime idea.

It is not the final product.

It is the first honest demonstration.

## What v001 Does Not Try To Prove Yet

For now, `v001` is **not** trying to prove:

- automatic dataset generation
- automatic dataset import from the internet
- an HTTP or LLM-facing server
- a full graph database
- a query language
- Wasm packaging
- fuzzy or semantic search

Those are good future expansions, but they are not the first checkpoint.

## The Practical Shape We Chose

The CLI surface is intentionally tiny:

```bash
knight-bus build --nodes-csv <path> --edges-csv <path> --output <dir>
knight-bus verify --snapshot <dir> --nodes-csv <path> --edges-csv <path>
knight-bus query --snapshot <dir> --entity <key> --dir forward|backward --hops <n> --format json|text
knight-bus bench --snapshot <dir> --report <dir>
```

The file shape stays aligned with the storage thesis:

```text
snapshot/
  manifest.json
  node_table.bin
  strings.bin
  forward.offsets.bin
  forward.peers.bin
  reverse.offsets.bin
  reverse.peers.bin
  key_index.bin
```

This is important because the plan did not change the storage idea.

It only changed how much of the system we are trying to prove in the first version.

## Tiny Example

Imagine we have:

```text
nodes.csv
  A
  B
  C

edges.csv
  A -> B
  A -> C
  C -> B
```

The build step turns that into a compact snapshot.

Then:

- `query forward A` should return `B, C`
- `query backward B` should return `A, C`

The important part is not that this tiny graph is impressive.

The important part is that the **same method** should work when the graph gets much larger.

## Why This Is The Highest-Probability Hackathon Path

This path is attractive because it removes confusion.

If the demo works, we know the core runtime idea works.

If the demo fails, we have a much smaller place to look.

That is exactly what a hackathon plan should do.

It should not try to win by hiding risk under more features.

It should win by making the core claim easy to test and easy to explain.
